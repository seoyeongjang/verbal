# 보안 규칙 테스트 계획

원문: `docs/SECURITY_RULES_TEST_PLAN.md`

## Firestore 허용 케이스

- 로그인 사용자는 사용자 프로필을 읽을 수 있습니다.
- 사용자는 유효한 필드와 아이디 정책으로 자신의 `users/{uid}` 프로필을
  생성/수정할 수 있습니다.
- 사용자는 자신의 `handles/{handle}` 선점을 생성/수정/삭제할 수 있습니다.
- 방 참여자는 자신의 방, 메시지, 멤버, 고정 메시지를 읽을 수 있습니다.
- 방 관리자는 참여 요청을 읽을 수 있습니다.
- 요청자는 자신의 참여 요청을 읽을 수 있습니다.

## Firestore 거부 케이스

- 익명 사용자는 앱 데이터를 읽거나 쓸 수 없습니다.
- 클라이언트는 방, 메시지, 멤버, 고정, 신고, usage, cooldown, transcription
  draft를 직접 생성할 수 없습니다.
- 사용자는 잘못된 아이디를 선점할 수 없습니다.
- 사용자는 다른 사용자의 프로필, 토큰, 아이디, 참여 요청을 쓸 수 없습니다.
- 방 참여자가 아닌 사용자는 방 데이터를 읽을 수 없습니다.

## Storage 허용 케이스

- 사용자는 자신의 voice draft를 업로드하고 읽을 수 있습니다.
- 방 참여자는 해당 방의 음성 메시지와 첨부파일을 읽을 수 있습니다.
- 클라이언트는 음성을 `voice_drafts/{uid}/...` 경로로만 업로드합니다.
  callable Functions가 방 참여자 검증 후 승인된 오디오를
  `voice_messages/{roomId}/...`로 복사합니다.
- 첨부 업로드 크기와 content type은 Storage rules와 일치해야 합니다.

## Storage 거부 케이스

- 익명 사용자는 파일을 읽거나 쓸 수 없습니다.
- 사용자는 다른 사용자의 draft 경로에 업로드할 수 없습니다.
- 클라이언트는 전송 완료된 음성 메시지 객체를 직접 쓸 수 없습니다.
- 사용자는 다른 사용자의 UID 경로에 첨부파일을 쓸 수 없습니다.
- 클라이언트는 전송 완료된 음성 메시지나 첨부파일을 직접 삭제할 수 없습니다.

## 런칭 전 필수

배포 대상 Storage rule 변경 뒤에는 `npm run rules:test`를 실행합니다.
현재 소스 rules와 운영 rules는 2026-05-28 기준 해당 테스트를 통과했습니다.
